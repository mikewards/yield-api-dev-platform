TBD Guides

Learn how to integrate TBD's DeFi yield APIs into your application. This guide covers everything from authentication to production deployment.

Authentication

All API requests require a Bearer token in the Authorization header:

Header 
Authorization: Bearer YOUR_API_KEY

API Keys

Create API keys in your dashboard. Each key is scoped to an application and environment (sandbox or production).

  • Sandbox keys start with tbd_sand_ and connect to testnets
  • Production keys start with tbd_prod_ and connect to mainnet

Base URL

Environment URL
Production https://flow-platform-production.up.railway.app
Sandbox https://flow-platform-flow-platform-staging.up.railway.app

Accounts

Your TBD account is your identity on the platform. Create one account, then create multiple applications under it.

Endpoint Description
POST/v1/accounts Create a new account
GET/v1/accounts/{id} Get account details
PATCH/v1/accounts/{id} Update account

Applications

Applications organize your projects. Each application gets its own Ethereum wallet and can have multiple API keys.

Endpoint Description
POST/v1/applications Create application
GET/v1/applications List applications
GET/v1/applications/{id} Get application
POST/v1/applications/{id}/tokens Create API key

Wallets

Each application automatically receives a segregated Ethereum wallet. Private keys are encrypted with AES-256-GCM.

Endpoint Description
POST/v1/applications/{id}/wallets Create additional wallet
GET/v1/applications/{id}/wallets List wallets
GET/v1/applications/{id}/wallets/{wallet_id} Get wallet details

Yield Accounts

Yield accounts hold your cryptocurrency and earn interest. Ground automatically routes deposits to the best available protocol (Morpho or Aave).

Supported Currencies

  • USDC, USDT, DAI (stablecoins)
  • ETH, WBTC (volatile assets)
Endpoint Description
POST/v1/yield/accounts Create yield account
GET/v1/yield/accounts List yield accounts
POST/v1/yield/accounts/{id}/deposit Deposit funds
POST/v1/yield/accounts/{id}/withdraw Withdraw funds
GET/v1/yield/rates Get current rates

Markets

View available markets across Morpho and Aave protocols with current yield rates.

Endpoint Description
GET/v1/markets List all markets
GET/v1/markets/{id} Get market details

Positions

Track your active yield positions, balances, and accrued interest.

Endpoint Description
GET/v1/yield/positions List all positions
GET/v1/yield/positions/{id} Get position details

Transactions

View transaction history including deposits, withdrawals, and yield accruals.

Endpoint Description
GET/v1/transactions List transactions
GET/v1/transactions/{id} Get transaction

Environments

Ground provides two environments for development and production use.

Sandbox

Use sandbox for development and testing. Connects to Ethereum testnets (Sepolia). No real funds required.

  • API keys start with tbd_sand_
  • Use testnet RPC URLs (Alchemy/Infura Sepolia)
  • Get testnet tokens from faucets

Production

Use production for live applications. Connects to Ethereum mainnet with real funds.

  • API keys start with tbd_prod_
  • Use mainnet RPC URLs
  • Real cryptocurrency deposits

Rate Limits

TBD enforces rate limits on all API endpoints to ensure fair usage and platform stability. Limits are configured per-endpoint and may vary based on the resource intensity of each operation.

Default Limits

Most endpoints have a default rate limit of 100 requests per minute per client.

Endpoint-Specific Limits

Endpoint Limit Window
GET /v1/markets 75 req 1 minute
GET /v1/yield/rates 75 req 1 minute
All other endpoints 100 req 1 minute
GET /health 300 req 1 minute

Rate Limit Headers

All API responses include rate limit information in the headers:

  • X-RateLimit-Limit — Maximum requests allowed per window
  • X-RateLimit-Remaining — Requests remaining in current window
  • X-RateLimit-Reset — Unix timestamp when the window resets

Handling Rate Limits

When you exceed a rate limit, the API returns a 429 Too Many Requests response:

Error Response 
{
  "status": "error",
  "code": 429,
  "message": "Rate limit exceeded. Please retry after 45 seconds.",
  "error": {
    "code": "RATE_LIMIT_EXCEEDED",
    "message": "You have exceeded the rate limit of 100 requests per minute.",
    "type": "rate_limit_error",
    "retry_after_seconds": 45
  }
}

The Retry-After header indicates how long to wait before retrying. Implement exponential backoff in your code to gracefully handle rate limits.

Webhooks

Receive real-time notifications when events happen in your account. Webhooks are delivered via Svix with automatic retries, signature verification, and delivery guarantees.

Event Types

Subscribe to the following events:

Event Description
deposit.completed Funds deposited to yield account
withdrawal.completed Funds withdrawn from yield account
yield.accrued Yield accrued to account (daily)
rate.changed Yield rate changed significantly (>0.5%)
account.status.changed Yield account status changed

Registering Endpoints

Register a webhook endpoint via the API or dashboard:

Create Webhook Endpoint 
curl https://flow-platform-production.up.railway.app/v1/webhooks \
  -X POST \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "url": "https://your-server.com/webhooks",
    "description": "Production webhook",
    "filter_types": ["deposit.completed", "withdrawal.completed"]
  }'

Webhook Payload

Webhook payloads are sent as POST requests with a JSON body. All payloads include application_id and application_name so you can identify which application triggered the event:

Example Payload 
{
  "event_type": "deposit.completed",
  "event_id": "evt_abc123def456",
  "timestamp": 1701456789000,
  "data": {
    "application_id": "app_abc123",
    "application_name": "Production App",
    "environment": "production",
    "yield_account_id": "ya_1234567890",
    "amount": "1000.00",
    "currency": "USDC",
    "protocol": "aave",
    "transaction_id": "txn_abc123"
  }
}

Note: If the request was made via dashboard (JWT auth) rather than an API key, application_id will be null.

Verifying Signatures

All webhooks include a signature header for verification. Use the Svix SDK to verify:

Node.js 
import { Webhook } from 'svix';

const wh = new Webhook(process.env.WEBHOOK_SECRET);

app.post('/webhooks', (req, res) => {
  try {
    const payload = wh.verify(req.body, req.headers);
    // Process the webhook
    console.log('Received event:', payload.event_type);
    res.status(200).send('OK');
  } catch (err) {
    console.error('Webhook verification failed:', err);
    res.status(400).send('Invalid signature');
  }
});

Retry Policy

Failed deliveries are automatically retried with exponential backoff:

  • Immediate retry on 5xx errors
  • Up to 5 retry attempts over 24 hours
  • Backoff intervals: 5min, 30min, 2hr, 8hr, 24hr
  • Endpoints disabled after consecutive failures

View delivery history and debug failed webhooks in your dashboard.

Best Practices

API Key Security

  • Never commit API keys to version control
  • Use environment variables for key storage
  • Rotate keys regularly
  • Use separate keys for sandbox and production

Error Handling

  • Implement retry logic with exponential backoff
  • Handle rate limits (429 responses)
  • Log all API errors for debugging
  • Set up webhook handlers for async errors

Production Checklist

  • Test thoroughly in sandbox first
  • Start with small amounts on mainnet
  • Monitor positions and transactions
  • Set up alerts for failures
  • Keep RPC keys secure
